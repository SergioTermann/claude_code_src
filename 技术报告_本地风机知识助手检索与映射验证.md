# 本地风机知识助手检索与风场机型映射验证技术报告

## 1. 背景

当前系统用于在本地环境中回答风机故障码、故障处理、设备原理以及风场与风机型号对应关系等问题。用户反馈的问题主要集中在以下几个方面：

- 本地模型调用效果不稳定，需要继续使用 LM Studio，但改用更小的本地模型。
- 用户问题并不总是故障码检索，也可能是设备原理咨询，需要系统自动识别问题类型。
- 故障码 `1100007` 曾被错误拆分成短码 `7`，导致命中错误资料。
- 风场与风机型号对应关系需要内置到系统中，避免依赖模糊检索造成误答。
- 需要将内置结果与文件夹中的标准资料进行抽样比对，确认当前问题点。

本报告基于当前代码和本地资料比对结果，对已完成改造、10 组数据验证结果、发现的问题以及后续整改方案进行说明。

## 2. 当前数据与代码范围

本次验证涉及的主要文件如下：

- 内置风场机型表：`src/data/windFarmModels.json`
- 风场机型查询工具：`src/utils/windFarmModels.ts`
- 本地 CLI 问答链路：`scripts/windrise-chat.mjs`
- LM Studio API 链路：`src/services/api/lmstudioClient.ts`
- 离线 smoke 测试：`scripts/smoke-llmwiki.mjs`
- LM Studio smoke 测试：`scripts/smoke-lmstudio.mjs`
- 标准化来源文件：`风机故障码/00 表达式规则涉及的要配置的标准化-型号和故障手册.md`
- 故障索引文件：`风机故障码/fault-index.jsonl`

其中，风场与风机型号对应关系的主校验依据为：

```text
风机故障码/00 表达式规则涉及的要配置的标准化-型号和故障手册.md
```

`fault-index.jsonl` 更适合用于故障资料检索索引，不适合作为风场与机型映射关系的唯一标准来源，因为其中的 `site` 字段经常存在多个风场合并、多个适用范围合并等情况。

## 3. 已完成改造

### 3.1 本地模型链路

当前方向保持为 LM Studio，不切回 Ollama。系统默认模型已调整为更小的本地模型，以降低本地调用压力，提高响应稳定性。

### 3.2 问题意图识别

系统已开始区分以下几类问题：

| 问题类型 | 示例 | 处理方式 |
| --- | --- | --- |
| 故障码咨询 | `1100007 是什么故障` | 优先走故障码检索 |
| 故障处理咨询 | `303804 怎么处理` | 走故障资料检索并输出处理建议 |
| 原理咨询 | `GSC 是什么原理` | 不强制进入故障码检索，走原理解释 |
| 风场机型查询 | `新华风电场对应什么风机` | 走内置风场机型映射表 |

该改造可以减少“原理问题被误当成故障码检索”的情况。

### 3.3 长数字故障码处理

已修复长数字故障码被错误拆分的问题。此前 `1100007` 曾被系统拆成尾号 `7`，导致命中华锐资料中的故障码 `7`，输出错误结果。

当前正确策略应为：

- 用户输入完整长故障码时，优先按完整故障码检索。
- 不自动把完整故障码降级成短码。
- 只有用户明确输入短码时，才允许短码检索。
- 输出故障码结果时必须带来源文件，便于人工核验。

### 3.4 内置风场机型映射

已新增内置映射数据文件：

```text
src/data/windFarmModels.json
```

并新增查询工具：

```text
src/utils/windFarmModels.ts
```

该工具支持：

- 按风场名查询机型。
- 按机型反查风场。
- 查询全部风场与机型对应关系。
- 通过别名匹配用户常用说法。
- 在 LM Studio 上下文中注入 `<风场机型映射>`，要求模型严格基于内置条目回答。

## 6. 发现的问题

### 6.1 主场站名不统一

当前部分内置数据的 `site` 字段使用了用户口语化名称，而不是标准文件中的场站名称。

典型问题如下：

| 当前主名称 | 标准名称 | 问题 |
| --- | --- | --- |
| 新华风电场 | 新华 | 多了“风电场” |
| 中溢B期风电场 | 中溢B期 | 多了“风电场” |
| 华能四平风电场一期风电场 | 华能四平风电场一期 | 重复“风电场” |
| 华能四平风电场二期风电场 | 华能四平风电场二期 | 重复“风电场” |
| 华能四平风电场二期/四期风电场 | 华能四平风电场二期/四期 | 末尾多了“风电场” |
| 华能四平风电场一期/四期风电场 | 华能四平风电场一期/四期 | 末尾多了“风电场” |

这会导致用户认为系统输出和文件夹中的标准结果不一致。

### 6.2 别名和主名称职责需要区分

当前部分用户常用说法被放在了 `site` 主字段中。更合理的结构应为：

- `site`：标准文件中的规范场站名，用于最终答案展示。
- `aliases`：用户可能输入的别名、长名称、俗称，用于匹配。

例如当前结构：

```json
{
  "site": "新华风电场",
  "aliases": ["新华"],
  "models": ["三一 SE8715", "华仪 HW2/S1500(87)", "运达 WD88-1500A"]
}
```

建议调整为：

```json
{
  "site": "新华",
  "aliases": ["新华风电场"],
  "models": ["三一 SE8715", "华仪 HW2/S1500(87)", "运达 WD88-1500A"]
}
```

这样用户输入“新华风电场”仍然可以命中，但系统输出会保持标准名称“新华”。

### 6.3 `fault-index.jsonl` 不适合作为机型映射标准

`fault-index.jsonl` 的用途是故障资料索引，其中的场站字段服务于故障码检索，不保证是一对一的场站机型关系。

例如某些故障资料可能适用于多个风场、多个机型或多个项目阶段，索引中会合并表达。如果直接用它校验风场机型映射，容易产生误判。

因此，风场机型关系应以标准化文件为准：

```text
风机故障码/00 表达式规则涉及的要配置的标准化-型号和故障手册.md
```

### 6.4 故障码检索仍需保持完整码优先

用户反馈的 `1100007` 错检索问题说明，故障码检索不能简单按短数字或尾号匹配。

后续应继续保持以下约束：

- 完整故障码优先级最高。
- 不允许自动把长故障码截断为短码。
- 对可能存在歧义的短码，应结合风场、品牌、机型共同限定。
- 如果用户没有提供风场或机型，系统应提示可能存在多个候选来源。

## 7. 建议整改方案

### 7.1 规范 `windFarmModels.json`

将 `src/data/windFarmModels.json` 中的 `site` 字段统一改为标准文件中的场站名。

用户常用名称、长名称、带“风电场”的名称全部放入 `aliases`。

示例：

```json
{
  "site": "华能四平风电场一期",
  "aliases": ["华能四平一期", "四平一期", "华能四平风电场一期风电场"],
  "models": ["金风 GW82-1500"]
}
```

### 7.2 输出结果统一使用标准场站名

用户输入：

```text
新华风电场对应什么风机
```

系统输出建议为：

```text
查询结果：
- 新华：三一 SE8715、华仪 HW2/S1500(87)、运达 WD88-1500A
```

不要输出：

```text
- 新华风电场：三一 SE8715、华仪 HW2/S1500(87)、运达 WD88-1500A
```

### 7.3 增加全量一致性测试

建议新增或扩展 smoke 测试，自动解析标准化文件，并与内置表进行全量比对。

测试项建议包括：

- 每条内置映射都能在标准化文件中找到对应来源。
- 每个标准场站下的品牌和型号集合与内置表一致。
- `site` 必须使用标准名称。
- `aliases` 必须支持用户常用说法。
- 用户输入长名称时，输出仍然使用标准名称。

### 7.4 加强意图分流测试

建议补充以下测试用例：

| 输入 | 期望路由 |
| --- | --- |
| `1100007是什么故障` | 故障码检索 |
| `故障码1100007怎么处理` | 故障码检索 |
| `GSC的作用是什么` | 原理咨询 |
| `变流器为什么会报过压` | 原理咨询 |
| `新华风电场对应哪些机型` | 风场机型映射 |
| `SE8715属于哪个风场` | 机型反查风场 |

## 8. 建议优先级

| 优先级 | 事项 | 原因 |
| --- | --- | --- |
| P0 | 修正 `windFarmModels.json` 的标准场站名 | 当前最容易造成用户感知错误 |
| P0 | 保持长故障码完整匹配优先 | 避免再次出现 `1100007` 误检索 |
| P1 | 增加风场机型全量一致性测试 | 防止后续维护时改坏映射 |
| P1 | 增加意图分流测试 | 防止原理问题误走故障码检索 |
| P2 | 优化无风场上下文下的歧义提示 | 提升复杂故障码咨询准确性 |

## 9. 结论

当前系统已经完成了风场机型关系内置、LM Studio 链路接入、故障码与原理咨询分流、长故障码完整匹配等关键改造。

通过 10 组抽样和 26 条全量机械比对可以确认：内置表中的机型对应关系本身是正确的，没有发现风场对应错误机型的问题。

当前最主要的问题是：部分 `site` 主显示名没有采用标准化文件中的规范场站名，而是使用了用户口语化名称或重复带有“风电场”的名称。该问题会影响答案展示的一致性，但不影响机型匹配的核心关系。

下一步建议优先修正 `windFarmModels.json` 中的主场站名，把用户常用说法放入 `aliases`，并增加自动化全量比对测试。完成后，系统既能兼容用户自然输入，又能输出与标准资料一致的结果。
